ACME

{{if eq .State "uninitialized"}}

ACME is available after setup. Initialize ShushTLS first.

{{else}}

ShushTLS acts as an ACME server (RFC 8555) so clients like acme.sh or certbot can obtain and renew certificates automatically. Challenges are not validated—the server responds positively to all challenges. Use this for automation when you control the network and trust is established by installing the root CA.

Directory URL

Configure your ACME client with this directory URL:

{{.BaseURL}}/acme/directory

acme.sh

Issue a certificate
acme.sh --issue -d example.local -w /var/www/html \
  --server {{.BaseURL}}/acme/directory

Use -w /path/to/webroot for webroot mode, or --standalone for local testing (requires socat). If the ShushTLS root CA is not installed on the client, add --insecure to skip TLS verification.

certbot

Issue a certificate
certbot certonly --webroot -w /var/www/html -d example.local \
  --server {{.BaseURL}}/acme/directory

Ensure the ShushTLS root CA is installed on the client, or certbot will reject the connection.

Discovery

To inspect the ACME directory:

curl -sS {{.BaseURL}}/acme/directory | jq .

What’s supported (RFC 8555 directory)

The directory lists only the endpoints ShushTLS implements. Omissions are intentional for a home/LAN CA, not bugs.

Directory fieldStatusNotes
newNonceImplementedReplay protection for POST requests.
newAccountImplementedRegister an ACME account (JWS-authenticated).
newOrderImplementedStart certificate issuance; challenges are auto-validated.
newAuthzNot implementedPre-authorization is unnecessary because challenges always succeed once an order is created.
revokeCertNot implementedOn a trusted LAN, “revocation” is re-issue under the same SAN (or stop using the old cert). No CRL/OCSP.
keyChangeNot implementedAccount key rotation is low value when the LAN is already trusted and compromise implies broader access.

If a stricter client refuses to proceed without omitted fields, use the REST API or file an issue.

Using with cert-manager

ShushTLS marks ACME challenges valid without contacting your cluster. That works for acme.sh and certbot, which post a response and trust the server. cert-manager is different: before it tells the ACME server to validate, it runs its own HTTP-01 self-check — a GET to http://{dnsName}/.well-known/acme-challenge/{token} from inside the cluster. For internal names (*.svc.cluster.local, bare hostnames like root, etc.) that self-check usually fails (no DNS, no ingress). ShushTLS would accept the challenge, but cert-manager never asks — the Certificate stays Pending.

Practical options (easiest first):

  1. REST API CronJob — skip ACME; fetch a fresh cert with GET /api/certificates/{san}?format=k8s-tls and kubectl apply -f - (see manifest below).
  2. DNS-01 webhook solver — cert-manager with a permissive DNS webhook (no propagation wait). Not bundled with ShushTLS today; track as a follow-up if you need native ACME inside Kubernetes.
  3. Make HTTP-01 self-check pass — ingress controller plus DNS rewrite so challenge URLs resolve to cert-manager’s solver (heavy for home/lab).

Example: CronJob via REST API

Replace SHUSHTLS_URL, credentials, SAN, and namespace. Requires the ShushTLS root CA trusted by the cluster or curl -k (not shown).

apiVersion: v1
kind: Secret
metadata:
  name: shushtls-api-auth
  namespace: default
type: Opaque
stringData:
  user: admin
  password: change-me
---
apiVersion: batch/v1
kind: CronJob
metadata:
  name: shushtls-cert-refresh
  namespace: default
spec:
  schedule: "0 3 * * 0"
  jobTemplate:
    spec:
      template:
        spec:
          restartPolicy: OnFailure
          containers:
            - name: refresh
              image: curlimages/curl:8.5.0
              command:
                - /bin/sh
                - -ec
                - |
                  curl -fsS -u "$(cat /auth/user):$(cat /auth/password)" \
                    "${SHUSHTLS_URL}/api/certificates/${CERT_SAN}?format=k8s-tls" \
                  | kubectl apply -f -
              env:
                - name: SHUSHTLS_URL
                  value: "https://shushtls.default.svc:8443"
                - name: CERT_SAN
                  value: "my-service.example.local"
              volumeMounts:
                - name: auth
                  mountPath: /auth
                  readOnly: true
          volumes:
            - name: auth
              secret:
                secretName: shushtls-api-auth

The CronJob container image must include kubectl or use an init pattern; swap in an image that has both curl and kubectl for production.
